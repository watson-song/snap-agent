# v0.5 Active Monitoring & Push ‚Äî Design Spec

**Date:** 2026-07-16
**Goal:** From "user asks, then diagnose" to "Agent proactively discovers issues and pushes"

## Architecture

v0.5 adds four new subsystems on top of v0.4's observability tools:

1. **HealthPatrolScheduler** ‚Äî periodically runs a diagnostic skill, stores results as patrol reports
2. **AnomalyEventListener SPI** ‚Äî event-driven skill triggering (host bridges Kafka/MQ ‚Üí SnapAgent)
3. **AlertConverger** ‚Äî in-memory alert deduplication and aggregation
4. **BugfixSuggester** ‚Äî transcript ‚Üí fix-suggestion generator

All four are gated by `snap-agent.patrol.enabled`, `snap-agent.alert.enabled`, etc. ‚Äî zero beans when disabled.

## 1. HealthPatrolScheduler

### Core SPI (`snap-agent-core`)

```java
package cn.watsontech.snapagent.core.patrol;

public interface PatrolScheduler {
    void schedule(PatrolTask task);
    void cancel(String patrolId);
    List<PatrolReport> getReports(String userId, int limit, int offset);
    long countReports(String userId);
}
```

**PatrolTask:**
- `id` ‚Äî unique patrol ID
- `skillName` ‚Äî skill to execute
- `cron` ‚Äî cron expression (Spring `CronExpression`)
- `inputs` ‚Äî Map<String, String> skill inputs
- `userId` ‚Äî user context for the patrol run
- `enabled` ‚Äî toggle without deleting

**PatrolReport:**
- `id`, `patrolId`, `taskId` (linked to AgentExecutor task)
- `skillName`, `triggeredAt`, `status` (RUNNING/SUCCESS/FAILED)
- `summary` ‚Äî short text extracted from transcript
- `anomalyDetected` ‚Äî boolean

### Starter implementation (`snap-agent-spring-boot-2x-starter`)

`ScheduledPatrolScheduler`:
- Uses Spring `TaskScheduler` (from `ThreadPoolTaskScheduler`)
- `schedule()` ‚Üí `TaskScheduler.schedule(runnable, CronTrigger)`
- Runnable: calls `AgentExecutor.execute()` with the skill + inputs, stores `PatrolReport`
- In-memory `PatrolReportStore` (ring buffer, default 500 entries)
- `getReports()` / `countReports()` ‚Äî pagination, sorted by triggeredAt desc

### Endpoints

| Endpoint | Description |
|----------|-------------|
| `POST /patrol/tasks` | Create a patrol task (cron + skill + inputs) |
| `GET /patrol/tasks` | List patrol tasks |
| `DELETE /patrol/tasks/{id}` | Delete a patrol task |
| `GET /patrol/reports?page=&size=` | List patrol reports (paginated) |
| `GET /patrol/reports/{id}` | Get a single patrol report |

### Config

```yaml
snap-agent:
  patrol:
    enabled: false
    scheduler-pool-size: 2
    report-buffer-size: 500
```

## 2. AnomalyEventListener SPI

### Core SPI

```java
package cn.watsontech.snapagent.core.patrol;

public interface AnomalyEventListener {
    void onEvent(AnomalyEvent event);
}
```

**AnomalyEvent:**
- `type` ‚Äî event type (e.g., "ERROR_SPIKE", "LATENCY_THRESHOLD", "CUSTOM")
- `source` ‚Äî source system (e.g., "order-service")
- `message` ‚Äî human-readable description
- `timestamp` ‚Äî epoch millis
- `metadata` ‚Äî Map<String, String> for extra context
- `skillName` ‚Äî which diagnostic skill to trigger (optional, defaults to "error-spike-investigation")
- `inputs` ‚Äî Map<String, String> skill inputs

### Default implementation

`DefaultAnomalyEventListener` (starter):
- Receives `AnomalyEvent`
- Triggers `AgentExecutor.execute()` with the specified skill + inputs
- Stores result as a patrol report (linked to the event)
- If `AlertConverger` is present, records the alert before triggering

### How host apps integrate

Host apps implement `AnomalyEventListener` or use the default and call it from their Kafka/RabbitMQ consumers:

```java
@Component
public class KafkaAnomalyBridge {
    @Autowired
    private AnomalyEventListener listener;

    @KafkaListener(topics = "anomaly-events")
    public void onMessage(ConsumerRecord<String, String> record) {
        AnomalyEvent event = parse(record.value());
        listener.onEvent(event);
    }
}
```

## 3. AlertConverger

### Core

```java
package cn.watsontech.snapagent.core.patrol;

public interface AlertConverger {
    AlertConvergence record(AnomalyEvent event);
    List<AlertConvergence> query(String userId, String type, int limit, int offset);
    long count(String userId, String type);
}
```

**AlertConvergence:**
- `id`, `fingerprint` ‚Äî hash(type + source + messageKeywords)
- `type`, `source`, `firstMessage`
- `count` ‚Äî number of occurrences
- `firstSeen`, `lastSeen` ‚Äî timestamps
- `status` ‚Äî ACTIVE / RESOLVED
- `relatedTaskId` ‚Äî linked patrol/diagnostic task

### Starter implementation

`InMemoryAlertConverger`:
- `ConcurrentHashMap<String, AlertConvergence>` keyed by fingerprint
- On `record()`: compute fingerprint ‚Üí if exists, increment count + update lastSeen; if new, create entry
- Ring buffer eviction (same pattern as InMemoryAuditStore)
- Auto-resolve: if no new occurrence within `auto-resolve-minutes`, mark RESOLVED

### Config

```yaml
snap-agent:
  alert:
    enabled: false
    buffer-size: 1000
    auto-resolve-minutes: 30
```

### Endpoints

| Endpoint | Description |
|----------|-------------|
| `GET /alerts?page=&size=&type=&status=` | List alerts (paginated, filterable) |
| `GET /alerts/{id}` | Get a single alert |
| `POST /alerts/{id}/resolve` | Manually resolve an alert |

## 4. BugfixSuggester

### Core SPI

```java
package cn.watsontech.snapagent.core.patrol;

public interface BugfixSuggester {
    BugfixSuggestion suggest(String taskId, List<TranscriptEvent> transcript);
}
```

**BugfixSuggestion:**
- `taskId` ‚Äî linked diagnostic task
- `rootCause` ‚Äî extracted root cause summary
- `affectedFiles` ‚Äî List<String> of file paths mentioned in transcript
- `suggestion` ‚Äî fix suggestion text (markdown)
- `confidence` ‚Äî LOW / MEDIUM / HIGH
- `commitRefs` ‚Äî List<String> of git commits mentioned

### Default implementation

`TemplateBugfixSuggester` (starter):
- Scans transcript for `tool_call` events where tool is `code_read` or `git_log`
- Extracts file paths from code_read results
- Extracts commit hashes from git_log results
- Extracts the last `response` event as root cause summary
- Generates a markdown suggestion template:
  ```
  ## Bugfix Suggestion

  ### Root Cause
  {extracted summary}

  ### Affected Files
  - {file1}
  - {file2}

  ### Suggested Fix
  Based on the root cause analysis, review the following files:
  - {file1}: Check the logic identified in the diagnostic
  - {file2}: Verify the changes mentioned in git history

  ### Related Commits
  - {commit1}

  ### Confidence: {level}
  ```
- Confidence: HIGH if both code_read and git_log were used, MEDIUM if only one, LOW if neither

### Endpoint

| Endpoint | Description |
|----------|-------------|
| `POST /runs/{id}/bugfix-suggestion` | Generate bugfix suggestion from a completed diagnostic run |

## Testing Strategy

- All SPIs tested via default implementations
- `PatrolReportStore` ‚Äî ring buffer eviction, pagination, sorting
- `InMemoryAlertConverger` ‚Äî fingerprint deduplication, count increment, auto-resolve
- `TemplateBugfixSuggester` ‚Äî transcript parsing, file extraction, confidence levels
- `ScheduledPatrolScheduler` ‚Äî mock TaskScheduler, verify schedule/cancel
- Controller endpoints ‚Äî ownership checks, pagination, error cases

---

## Design Principles

1. **SPI-first** ‚Äî all four subsystems have interfaces in core, implementations in starter
2. **Zero impact when disabled** ‚Äî `snap-agent.patrol.enabled=false` ‚Üí no beans
3. **In-memory only** ‚Äî no database, no external dependencies
4. **Read-only** ‚Äî patrol runs are diagnostic (read-only skills), alert convergence is passive
5. **Host-extensible** ‚Äî `AnomalyEventListener` and `BugfixSuggester` can be overridden by host beans
