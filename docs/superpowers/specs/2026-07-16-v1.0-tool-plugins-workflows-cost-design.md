# v1.0 — Tool Plugins, Workflows, Cost Accounting Design

## Goal

Build the three final subsystems to complete the v1.0 roadmap: tool plugin SPI, workflow orchestration, and LLM cost tracking.

## Architecture

All three subsystems follow the established SPI-first pattern: interfaces in core module, default implementations in starter. Each is gated by `@ConditionalOnProperty(enabled=false)` for zero impact when not configured.

---

## 1. Tool Plugin SPI

### Core SPI

**`ToolPlugin`** (core) — groups multiple `ToolProvider` instances as a logical plugin unit:

```java
public interface ToolPlugin {
    String pluginId();                          // unique plugin identifier
    String description();                       // human-readable description
    List<ToolProvider> toolProviders();          // tools provided by this plugin
}
```

### Starter Implementation

**`SimpleToolPluginRegistry`** — collects all `ToolPlugin` beans, exposes `listPlugins()` for introspection. Each plugin's `ToolProvider` instances are already auto-discovered by Spring's `@Component` scanning or explicit `@Bean` declarations. The registry provides metadata — not a separate wiring path.

**`ToolPluginInfo`** — DTO with pluginId, description, tool names list.

### Config

```yaml
snap-agent:
  tool-plugins:
    enabled: false   # enables plugin introspection endpoint
```

### Controller Endpoint

`GET /tool-plugins` — lists registered plugins and their tools (read-only introspection).

## 2. Workflow Orchestration

### Core SPI

**`Workflow`** — immutable definition:

```java
public interface Workflow {
    String getId();
    String getDescription();
    List<WorkflowStep> getSteps();
}
```

**`WorkflowStep`** — a single step in the workflow:

```java
public class WorkflowStep {
    String name;
    String skillId;                    // skill to execute
    Map<String, String> inputs;         // static inputs + ${prev.result.field} refs
    String condition;                   // optional SpEL-like condition (null = always)
    WorkflowStepFailureStrategy onFailure; // ABORT, SKIP, RETRY
}
```

**`WorkflowStepFailureStrategy`** — enum: ABORT, SKIP, RETRY

**`WorkflowExecutor`** (SPI) — runs a workflow:

```java
public interface WorkflowExecutor {
    WorkflowResult execute(Workflow workflow, String userId, String model, Map<String, String> triggerInputs);
}
```

**`WorkflowResult`** — step results + final status:

```java
public class WorkflowResult {
    String workflowId;
    Map<String, StepResult> stepResults;   // stepName → result
    WorkflowStatus status;                 // RUNNING, COMPLETED, ABORTED, FAILED
    String summary;
}
```

**`StepResult`** — captures task ID, status, report from each step.

**`WorkflowStatus`** — enum: RUNNING, COMPLETED, ABORTED, FAILED

### Starter Implementation

**`SimpleWorkflowExecutor`** — sequential step execution:
1. Resolve step inputs (replace `${step.result.field}` references with prior step results)
2. Execute skill via `AgentExecutor.execute()` with `AgentTask.create()`
3. Check task status: SUCCEEDED → continue, FAILED → apply onFailure strategy
4. Store step result
5. Return aggregated `WorkflowResult`

Input resolution: simple pattern matching — `${stepName.result}` → full report text, `${stepName.taskId}` → task ID. No complex expression engine for v1.0.

**`WorkflowRegistry`** — in-memory, loaded from YAML config:

```yaml
snap-agent:
  workflows:
    enabled: false
    definitions:
      - id: full-diagnose
        description: "全链路诊断工作流"
        steps:
          - name: health-check
            skill: health-check
            inputs:
              service: "${trigger.service}"
            onFailure: ABORT
          - name: find-root-cause
            skill: error-spike-investigation
            condition: "${health-check.status == 'SUCCEEDED'}"
            inputs:
              timeWindow: "1h"
            onFailure: SKIP
          - name: suggest-fix
            skill: code-analysis
            onFailure: SKIP
```

### Config Properties

`Workflows` inner class in `SnapAgentProperties`:
- `enabled` (boolean, default false)
- `definitions` (List<WorkflowDef>)

`WorkflowDef` inner class:
- `id`, `description`
- `steps` (List<StepDef>)

`StepDef` inner class:
- `name`, `skill`, `inputs` (Map), `condition`, `onFailure` (default ABORT)

## 3. Cost Accounting

### Core SPI

**`CostTracker`** — records cost per LLM call:

```java
public interface CostTracker {
    void record(CostRecord record);
    CostSummary getSummary(String userId, String skillName, long from, long to);
    List<CostRecord> getRecords(String userId, long from, long to);
}
```

**`CostRecord`** — single cost event:

```java
public class CostRecord {
    String userId;
    String skillName;
    String taskId;
    String model;
    long inputTokens;
    long outputTokens;
    long cacheReadTokens;
    double cost;          // calculated cost in configured currency
    long timestamp;
}
```

**`CostSummary`** — aggregated summary:

```java
public class CostSummary {
    String dimension;        // "user", "skill", "global"
    String dimensionValue;
    double totalCost;
    long totalInputTokens;
    long totalOutputTokens;
    long totalCacheReadTokens;
    int requestCount;
    double budget;           // 0 if no budget configured
    double utilization;      // budget > 0 ? totalCost / budget : 0
}
```

### Starter Implementation

**`InMemoryCostTracker`** — `List<CostRecord>` in synchronized storage. Computes summaries by filtering and aggregating on demand. `@ConditionalOnMissingBean` allows host to provide DB-backed implementation.

**`CostCalculator`** — converts token counts to monetary cost based on configured pricing:

```java
public class CostCalculator {
    double inputPricePerMillion;   // e.g., 3.00
    double outputPricePerMillion;  // e.g., 15.00
    double cacheReadPricePerMillion; // e.g., 0.30 (discounted)
    String currency;

    double calculate(long inputTokens, long outputTokens, long cacheReadTokens);
}
```

### Integration

Cost tracking hooks into `AgentExecutor` via a new `CostAwareLlmEventSink` wrapper or a callback on `LlmEventSink.onStop()`. Since the LLM client already reports `stopReason`, we add an `onUsage` callback to `LlmEventSink`:

```java
// LlmEventSink — new default method
default void onUsage(long inputTokens, long outputTokens, long cacheReadTokens) {}
```

The `AnthropicLlmClient` already receives usage data in the SSE stream. When usage data arrives, it calls `onUsage()`. The `TurnCollector` in `AgentExecutor` captures these totals and, at task completion, calls `CostTracker.record()`.

### Config

```yaml
snap-agent:
  cost:
    enabled: false
    pricing:
      input-per-million: 3.00
      output-per-million: 15.00
      cache-read-per-million: 0.30
      currency: CNY
    budgets:
      per-user-daily: 10.00
      per-skill-daily: 50.00
      global-daily: 500.00
```

### Controller Endpoints

- `GET /cost/summary?from=&to=&groupBy=user|skill` — aggregated cost summary
- `GET /cost/users/{userId}/summary?from=&to=` — per-user breakdown
- `GET /cost/skills/{skillName}/summary?from=&to=` — per-skill breakdown

## File Structure

### Core SPI (snap-agent-core)

- `core/plugin/ToolPlugin.java` — plugin interface
- `core/plugin/ToolPluginInfo.java` — plugin info DTO
- `core/workflow/Workflow.java` — workflow definition SPI
- `core/workflow/WorkflowStep.java` — step model
- `core/workflow/WorkflowStepFailureStrategy.java` — enum
- `core/workflow/WorkflowExecutor.java` — executor SPI
- `core/workflow/WorkflowResult.java` — result model
- `core/workflow/StepResult.java` — per-step result
- `core/workflow/WorkflowStatus.java` — status enum
- `core/cost/CostTracker.java` — cost tracker SPI
- `core/cost/CostRecord.java` — cost record model
- `core/cost/CostSummary.java` — cost summary model

### Starter Implementations (snap-agent-spring-boot-2x-starter)

- `boot2x/plugin/SimpleToolPluginRegistry.java` — collects ToolPlugin beans
- `boot2x/workflow/SimpleWorkflowExecutor.java` — sequential step runner
- `boot2x/workflow/WorkflowRegistry.java` — loads from config
- `boot2x/cost/InMemoryCostTracker.java` — in-memory cost storage
- `boot2x/cost/CostCalculator.java` — token→cost calculation
- `boot2x/cost/CostAwareExecutorHook.java` — hooks into AgentExecutor

- Add `ToolPlugins`, `Workflows`, `Cost` inner classes to properties
