## 1. 目标

从"一次性诊断"升级为"问题→诊断→方案→修复→验证→沉淀"的完整闭环。

本次交付范围：
- `SolutionSuggester` SPI：基于诊断结果生成候选解决方案
- `IssueTracker` SPI：问题追踪（创建/更新/查询 Issue）
- `VerificationRunner` SPI：修复后验证回归
- `IssueClosure` 数据模型：闭环全生命周期记录
- `TemplateSolutionSuggester`：规则模板方案生成器
- `InMemoryIssueStore`：内存问题存储
- `SimpleVerificationRunner`：重跑诊断 skill + 状态对比
- REST 端点：方案/Issue/验证/关闭
- 配置：`snap-agent.issue-tracking.*`

不在本次范围（延后到 v0.9.1+）：
- Jira/GitHub Issues 原生对接（需各平台 SDK）— v0.9.1
- 自动 diff 生成（需 LLM 辅助）— v0.9.1
- 知识库自动入库（需确认流程 + 摘要能力）— v0.9.1

## 3. Core 层数据模型与 SPI

### 3.1 IssueClosure

```java
public class IssueClosure {
    private String id;
    private String conversationId;      // 关联诊断会话
    private String taskId;              // 关联 Agent 任务
    private String rootCauseSummary;    // 根因摘要
    private SolutionSuggestion solution;// 方案建议
    private IssueStatus status;         // 状态枚举
    private String externalIssueId;     // 外部 Issue ID (Jira/工单)
    private VerificationResult verification; // 验证结果
    private String knowledgeEntryId;    // 沉淀到知识库的条目 ID
    private long createdAt;
    private long updatedAt;
}
```

状态枚举:
```
DIAGNOSED → SOLUTION_PROPOSED → ISSUE_CREATED → FIX_IN_PROGRESS → VERIFIED → CLOSED
```

### 3.2 SolutionSuggestion

```java
public class SolutionSuggestion {
    private String id;
    private List<SolutionOption> options;  // 候选方案列表
    private String recommendedOptionId;     // 推荐方案 ID
    private String relatedCode;             // 相关代码位置
    private String rationale;               // 推荐理由
}

public class SolutionOption {
    private String id;
    private String title;
    private String description;
    private String effort;  // "low" / "medium" / "high"
    private boolean temporary; // 临时方案 or 长期修复
}
```

### 3.3 VerificationResult

```java
public class VerificationResult {
    private String id;
    private boolean passed;
    private String summary;
    private String beforeStatus;  // 修复前状态
    private String afterStatus;   // 修复后状态
    private long verifiedAt;
}
```

### 3.4 SPI 接口

```java
public interface SolutionSuggester {
    SolutionSuggestion suggest(IssueClosure issue, String transcriptSummary);
}

public interface IssueTracker {
    String createIssue(IssueClosure issue);
    IssueClosure getIssue(String issueId);
    void updateStatus(String issueId, IssueStatus status);
    List<IssueClosure> listIssues(int limit, int offset);
}

public interface VerificationRunner {
    VerificationResult verify(IssueClosure issue);
}
```

## 4. Starter 层实现

### 4.1 TemplateSolutionSuggester

基于规则模板的方案生成。根据 transcript 中的工具调用和根因关键词，匹配预定义模板。

模板示例:
- 根因含"参数缺失" → 方案1: 手动插入参数 + 方案2: 修改过滤条件
- 根因含"连接超时" → 方案1: 检查连接池配置 + 方案2: 增加超时时间
- 根因含"数据为空" → 方案1: 检查上游任务 + 方案2: 补数据
- 默认模板 → 方案1: 根据代码图谱定位修复点 + 方案2: 联系负责人确认

### 4.2 InMemoryIssueStore

`IssueTracker` 的内存实现:
- `Map<String, IssueClosure>` 存储
- `createIssue` 生成 ID, 存入, 返回 ID
- `listIssues` 按 createdAt 降序分页

### 4.3 SimpleVerificationRunner

重跑诊断 skill + 状态对比:
- 获取 `IssueClosure` 关联的 taskId
- 从 transcript 提取最后一个工具调用的结果作为 "before"
- 重新执行同一 skill + 同一 inputs → 获取 "after"
- 对比 before/after → 生成 VerificationResult

## 5. 配置树

```yaml
snap-agent:
  issue-tracking:
    enabled: false           # 闭环总开关
    auto-suggest: true       # 诊断完成后自动生成方案建议
    auto-verify: false       # 修复后自动验证 (需重跑 skill)
```

## 6. REST 端点

| 端点 | 说明 |
|------|------|
| `POST /runs/{id}/solution` | 获取/生成解决方案建议 |
| `POST /runs/{id}/issue` | 创建 Issue (关联诊断会话) |
| `GET /issues` | 列出所有 Issue |
| `GET /issues/{id}` | 查看 Issue 闭环状态 |
| `POST /issues/{id}/verify` | 触发验证 |
| `POST /issues/{id}/close` | 关闭并沉淀经验 |
