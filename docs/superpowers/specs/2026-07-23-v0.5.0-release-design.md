# v0.5.0 发布流程设计

**日期**: 2026-07-23
**状态**: Approved
**作者**: brainstorming session

## 背景与问题

代码库当前状态:
- **pom.xml 版本**: `0.4.0-feat-v0.5-plugin-improvement-SNAPSHOT` (非标准 SNAPSHOT, 带分支限定符, 来自 commit `17f3f24` 的 v0.5 Plugin Architecture Refactor)
- **最新 git tag**: `v0.3.0` (最后一个正式发布版)
- **MEMORY.md 路线图**: v0.1-v1.1 标记为"全部交付完成", 但 git tag 只到 v0.3.0 — 说明功能已开发但未正式打 tag 发布
- **分支状态**: `main` 领先 `dev` 20 个 commit, `dev` 完全包含在 `main` 里(包括最近的 SPI 工作和 spec/plan 文档)
- **docs 版本引用**: 多处引用旧版本号(0.3.0/0.4.0), 与实际代码状态脱节

## 目标

1. 把当前代码状态正式发布为 `v0.5.0` 到 Maven Central + GitHub Releases
2. 跳过 v0.4.0 (代码已远超 0.4.0 范围, 发 0.4.0 反而失真)
3. 走 `./scripts/release.sh` 标准发版流程, 在 `main` 分支上执行
4. 同步更新所有 live 文档的版本引用 (不动历史快照 specs/plans)
5. 发布后自动切到 `0.6.0-SNAPSHOT` 作为下一开发版本
6. 走 CI 发布 (推 tag 后 GitHub Actions 自动发布到 Maven Central + GitHub Release)

## 非目标

- 不动 `dev` 分支 (留待后续按需 `git checkout dev && git merge main`)
- 不打 v0.4.0 tag (跳过)
- 不动 demo 模块功能代码 (只动版本号)
- 不改 CI workflow 文件 (`.github/workflows/*.yml`)
- 不改历史 specs/plans 文档 (`docs/superpowers/specs/*`, `docs/superpowers/plans/*`)

## 设计

### 整体流程 (顺序执行)

```
[当前 main HEAD: a1250ed]
  ↓
A. 更新 live docs 版本引用 → commit "docs: bump version references to 0.5.0"
  ↓
B. ./scripts/release.sh 0.5.0 --next 0.6.0  (在 main 上跑)
   内部步骤:
   - mvn versions:set 改 parent + 3 个模块 (snap-agent-core/starter/client) 到 0.5.0
   - sed 改 snap-agent-demo 到 0.5.0
   - mvn clean verify (含 jacoco 覆盖率检查)
   - git commit "release: v0.5.0"
   - git tag v0.5.0
   - 改回 0.6.0-SNAPSHOT + git commit "chore: bump to 0.6.0-SNAPSHOT"
  ↓
C. git push origin main && git push origin v0.5.0
  ↓
D. CI 自动触发 (tag v*):
   - maven-central.yml: GPG 签名 + 发布到 Maven Central (autoPublish)
   - release.yml: 创建 GitHub Release (jar + standalone pom + install.sh)
  ↓
E. 验证 (15-30 分钟后):
   - https://central.sonatype.com/artifact/cn.watsontech.snapagent/snap-agent-core
   - https://repo1.maven.org/maven2/cn/watsontech/snapagent/
```

### Live docs 范围 (A 步)

| 类别 | 文件 | 改什么 |
|------|------|--------|
| 发版指南 | `docs/RELEASING.md` | 示例命令 `0.3.0 --next 0.4.0` → `0.5.0 --next 0.6.0` |
| 路线图 | `docs/ROADMAP.md` | 标记 v0.5.0 已发布, 下一 dev 版本 `0.6.0-SNAPSHOT` |
| 集成文档 | `docs/embeed-skills-agent/{07,09,10}.md` | `<version>0.x.x</version>` 引用 → `0.5.0` |
| 网站文档 | `docs/site/**` (架构/手册/集成/插件) | 版本号引用 → `0.5.0` 或 "latest" |
| README | `README.md` / `README_zh.md` | 版本徽章 / `<version>` 标签 → `0.5.0` |

**不动**: `docs/superpowers/specs/*` 和 `plans/*` (历史快照, 改了失真)

**策略**:
- 面向用户文档 (README, site/manual) 优先用 "latest" / "current" 替代具体版本号, 减少后续每次发版的维护成本
- 必须写死版本的地方 (Maven `<version>` 标签示例, 发版命令示例) 才写 `0.5.0`

### 关键决策点

- **Next dev version**: `0.6.0-SNAPSHOT` (MEMORY 路线图 v0.6 是 platformization, 自然衔接)
- **跳过 0.4.0**: pom 当前是 `0.4.0-feat-...-SNAPSHOT`, 直接发 0.5.0 (代码已远超 0.4.0 范围)
- **CI vs 本地**: 走 CI (推 tag 后 GitHub Actions 自动发布), 依赖 GitHub Secrets:
  - `CENTRAL_USERNAME` / `CENTRAL_PASSWORD` (Portal User Token)
  - `GPG_PRIVATE_KEY` / `GPG_PASSPHRASE`
  - 上次 v0.3.0 发过, 应该已配; 如未配, 走 `--deploy-local` 本地发布 (需本机 GPG + settings.xml)
- **在 main 上跑**: 偏离 RELEASING.md "推 dev" 的历史惯例, 但 CI 只看 tag 不看分支; 事后想对齐 dev 只需 `git checkout dev && git merge main`

### 失败回滚

| 失败点 | 处置 |
|--------|------|
| A 步 docs 改错 | 改 docs, 不涉及不可逆操作 |
| B 步 `mvn verify` 测试失败 | 不 commit 不 tag, 修测试后重跑 release.sh |
| B 步 commit/tag 已建但还没推 | `git tag -d v0.5.0 && git reset --hard HEAD~2` 撤销 (本地, 无影响) |
| C 步已推送 tag, 但 CI 还没发布 | `git push origin :refs/tags/v0.5.0` 删远端 tag + `git reset --hard origin/main` 撤 main。**前提: CI 还没发到 Central** |
| D 步 CI 已发布到 Central | **不可撤销**, 只能发新版本号 (0.5.1 或 0.6.0) |
| D 步 CI 发布失败 | GitHub Actions 日志查原因, 改了后**不能用同版本号重发**, 必须 0.5.1 或 0.6.0 |

### 测试

- A 步 docs 改动不涉及代码, 无需测试
- B 步 release.sh 内部跑 `mvn clean verify` (含 jacoco), 已覆盖测试验证
- 发布后人工验证: Maven Central 拉取 + GitHub Release 页面可见

## 实现步骤概览

1. 扫描并更新所有 live docs 版本引用 (A 步)
2. 跑 `./scripts/release.sh 0.5.0 --next 0.6.0` (B 步)
3. 推 main + tag (C 步)
4. 监控 CI workflow 执行 (D 步)
5. 验证 Maven Central + GitHub Release (E 步)
6. (可选) 同步 dev 分支: `git checkout dev && git merge main`
